## 1. Jenis Kegiatan YYZU
Kegiatan di YYZU dibagi menjadi tiga klaster berdasarkan sifat dan tujuannya:
### Klaster A: Learning & Sharing
Kegiatan berbasis pengetahuan yang tidak menghasilkan deliverable teknis.
| Kegiatan | Durasi | Frekuensi | PIC Default |
| --- | --- | --- | --- |
| **Study Group** | 60-120 menit | Sesuai kebutuhan | Divisi yang relevan |
| **Workshop** | 2-4 jam | Per batch | Divisi/Mentor |
| **Industry Sharing** | 60-90 menit | Bulanan | BPH/Divisi |
| **Career Preparation** | 20-90 menit | Per batch | Divisi SDM |
| **Technical Discussion** | 30-60 menit | Sesuai kebutuhan | Inisiator |
### Klaster B: Review & Simulation
Kegiatan berbasis praktik terbatas menggunakan artefak nyata.
| Kegiatan | Durasi | Frekuensi | PIC Default |
| --- | --- | --- | --- |
| **Code Review (standalone)** | 30-60 menit | Sesuai kebutuhan | Reviewer |
| **Design Review (standalone)** | 30-60 menit | Sesuai kebutuhan | Reviewer |
| **Portfolio Review** | 30-45 menit per member | Per batch | Divisi SDM/Mentor |
| **Workflow Simulation** | 60-120 menit | Per batch | Divisi PM |
### Klaster C: Showcase & Community
Kegiatan yang memamerkan hasil kerja dan memperkuat komunitas.
| Kegiatan | Durasi | Frekuensi | PIC Default |
| --- | --- | --- | --- |
| **Demo Day** | 2-3 jam | Akhir batch | Divisi PM + Event |
| **Batch Showcase** | 1-2 jam | Akhir batch | Event + Media |
| **Community Meetup** | 2-3 jam | Per semester | Event + Partnership |
| **Onboarding Session** | 1-2 jam | Awal batch | Divisi SDM + Event |

## 2. SOP Perencanaan Event
Setiap event wajib mengikuti tahapan berikut:
### Tahap 1: Ideasi & Proposal (H-21 sampai H-14)
1. PIC mengajukan proposal event ke Koordinator Umum berisi:
	- Tujuan dan target peserta
	- Jenis kegiatan (Klaster A/B/C)
	- Timeline persiapan
	- Kebutuhan sumber daya (venue, tools, narasumber)
	- Estimasi budget (jika ada)
2. Koordinator Umum menyetujui atau meminta revisi dalam 3 hari kerja.
3. Jika disetujui, PIC membuat task di project board.
### Tahap 2: Persiapan (H-14 sampai H-3)
1. PIC menyusun rundown detail event.
2. PIC mengkoordinasikan kebutuhan dengan divisi terkait:
	- **Media & Branding**: materi publikasi, desain poster
	- **Partnership**: undangan narasumber eksternal (jika ada)
	- **Learning & Curriculum**: materi teknis (jika workshop/study group)
3. PIC memastikan venue/platform sudah dikonfirmasi.
4. PIC menyebarkan undangan minimal H-5.
### Tahap 3: Pelaksanaan (H-0)
1. PIC melakukan pengecekan final 1 jam sebelum event.
2. PIC memastikan:
	- Materi presentasi sudah siap
	- Venue/platform sudah aktif
	- Narasumber sudah konfirmasi kehadiran
	- Dokumentasi sudah disiapkan (screen recording, foto)
3. PIC membuka dan menutup sesi sesuai rundown.
4. PIC mencatat hal-hal penting selama sesi berlangsung.
### Tahap 4: Evaluasi (H+1 sampai H+7)
1. PIC mengirimkan survei kepuasan peserta (maksimal 5 pertanyaan).
2. PIC menyusun laporan evaluasi event berisi:
	- Jumlah peserta aktual vs target
	- Skor kepuasan rata-rata
	- Temuan dan rekomendasi perbaikan
	- Dokumentasi (link rekaman, foto, materi)
3. Laporan diserahkan ke Koordinator Umum dan diarsipkan di Notion.

## 5. Evaluasi & Pelaporan
### Metrik Keberhasilan Event
| Metrik | Target | Cara Ukur |
| --- | --- | --- |
| Kehadiran peserta | Minimal 70% dari target | Absensi |
| Skor kepuasan | Minimal 4/5 | Survei pasca-event |
| Kelengkapan dokumentasi | 100% | Checklist |
| Ketepatan waktu | Maksimal 15 menit delay | Catatan PIC |
### Pelaporan Berkala
Divisi Event & Planning wajib menyerahkan:
- **Laporan per-event**: H+7 setelah event
- **Ringkasan batch**: di akhir setiap batch, berisi rekap seluruh event yang dilaksanakan

## 7. Budget & Logistik
### Template Budget Event
```markdown
# Budget Event: [Nama Event]

| Item | Estimasi | Aktual | Keterangan |
|---|---|---|---|
| Venue/Platform | Rp | Rp | |
| Snack/Konsumsi | Rp | Rp | Jumlah peserta x harga/pax |
| Merchandise/Doorprize | Rp | Rp | Opsional |
| Cetak (poster, sertifikat) | Rp | Rp | |
| Transport narasumber | Rp | Rp | Jika offline |
| Lain-lain | Rp | Rp | |
| **TOTAL** | **Rp** | **Rp** | |
```
### Mekanisme Pengajuan Budget
1. PIC mengisi template budget dan mengajukan ke Koordinator Umum bersamaan dengan proposal event.
2. Koordinator Umum menyetujui atau meminta penyesuaian dalam 3 hari kerja.
3. Budget di atas Rp25.000.000 wajib mendapat persetujuan DWA.
4. Setelah event, PIC menyerahkan laporan realisasi budget (estimasi vs aktual).
### Checklist Logistik
- [ ] Venue/platform sudah dikonfirmasi
- [ ] Konsumsi sudah dipesan (jika offline)
- [ ] Alat presentasi sudah disiapkan (proyektor, speaker, mic)
- [ ] Merchandise/doorprize sudah disiapkan (jika ada)
- [ ] Sertifikat sudah disiapkan (jika ada)
- [ ] Dokumentasi sudah disiapkan (kamera, screen recording)

## 8. Handling Narasumber
### Template Undangan Narasumber
```markdown
Subject: Undangan Narasumber - [Nama Event] YYZU

Yth. Bapak/Ibu [Nama],

Kami dari YYZU (Collaborative Technology Ecosystem) ingin mengundang Bapak/Ibu sebagai narasumber dalam kegiatan kami:

📅 Tanggal: [DD/MM/YYYY]
⏰ Waktu: [HH:MM - HH:MM WIB]
📍 Platform: [Online/Offline + Detail]
👥 Peserta: [Jumlah + segmentasi]
🎤 Topik: [Topik yang diharapkan]

[Konteks singkat tentang YYZU dan tujuan event]

Apakah Bapak/Ibu berkenan untuk berpartisipasi? Kami sangat menghargai kontribusi Bapak/Ibu dalam pengembangan talenta teknologi Indonesia.

Terima kasih.

Hormat kami,
[Nama PIC]
Divisi Event & Planning YYZU
[Email/WhatsApp]
```
### Checklist Konfirmasi Narasumber
- [ ] Kirim undangan minimal H-14
- [ ] Follow-up jika belum ada respons dalam 3 hari
- [ ] Konfirmasi ulang H-3 sebelum event
- [ ] Kirim rundown dan teknis presentasi H-2
- [ ] Kirim reminder H-1
- [ ] Siapkan narasumber cadangan jika batal
### SOP Pembatalan Narasumber
1. Jika narasumber batal H-3 atau lebih, PIC segera mencari pengganti.
2. Jika narasumber batal H-1 atau H-0, PIC menginformasikan peserta dan menjadwal ulang atau mengganti format event.
3. PIC melaporkan pembatalan ke Koordinator Umum.

## 9. Platform & Tools
### Platform yang Digunakan
| Jenis Event | Platform Utama | Alternatif | Cara Setup |
| --- | --- | --- | --- |
| **Online - Webinar** | Google Meet | Zoom | Buat link di Google Calendar, share ke peserta |
| **Online - Workshop** | Discord (voice channel) | Google Meet | Buat voice channel khusus, share screen |
| **Online - Study Group** | Discord (voice channel) | - | Gunakan channel yang sudah ada |
| **Offline - Workshop** | Venue kampus/ruangan | - | Konfirmasi venue H-7 |
| **Offline - Meetup** | Kafe/coworking space | - | Survey venue H-14 |
### Checklist Teknis Platform Online
- [ ] Link meeting sudah dibuat dan diuji
- [ ] Waiting room / lobby sudah dikonfigurasi
- [ ] Screen sharing sudah diuji
- [ ] Recording sudah diaktifkan (jika diperlukan)
- [ ] Backup platform sudah disiapkan (jika utama gagal)
- [ ] Moderator sudah ditunjuk untuk mengelola chat
### Troubleshooting Dasar
| Masalah | Solusi |
| --- | --- |
| Narasumber tidak bisa join | Kirim link alternatif via WhatsApp |
| Audio tidak keluar | Minta peserta cek microphone dan speaker |
| Screen sharing gagal | Minta narasumber restart aplikasi |
| Koneksi internet putus | Pindah ke platform alternatif |
| Peserta terlalu banyak | Pindah ke platform dengan kapasitas lebih besar |

## 10. Risk Management
### Daftar Risiko Umum
| Risiko | Dampak | Mitigasi |
| --- | --- | --- |
| Narasumber batal mendadak | Event batal atau kualitas menurun | Siapkan narasumber cadangan, miliki materi backup |
| Teknis platform gagal | Peserta tidak bisa join | Siapkan platform alternatif, lakukan test H-1 |
| Peserta sedikit (\< 50% target) | Dampak event berkurang | Perkuat promosi, perpanjang deadline pendaftaran |
| Materi tidak siap | Sesu kosong atau terburu-buru | Minta materi dari narasumber minimal H-3 |
| Konflik jadwal dengan event lain | Peserta terpecah | Cek kalender ekosistem sebelum menetapkan jadwal |
| Hujan/bencana (offline) | Peserta tidak datang | Siapkan rencana pindah ke online |
### SOP Penanganan Darurat
1. **Narasumber batal H-0:** PIC segera menginformasikan peserta, menawarkan alternatif (reschedule atau format diskusi).
2. **Platform down:** Moderator menginformasikan peserta untuk pindah ke platform alternatif.
3. **Peserta \< 30% target:** Event tetap dilaksanakan, PIC mencatat untuk evaluasi promosi.

## 13. Kalender Event Batch
Kalender event adalah panduan bagi Divisi Event & Planning dalam merencanakan kegiatan sepanjang siklus batch. Setiap batch memiliki event wajib (milestone) dan event opsional.
Untuk kalender lengkap, template penjadwalan, dan aturan penjadwalan, lihat sub-halaman: [📆 Kalender Event Batch YYZU](/docs/operations/event-calendar).
### Ringkasan Event Wajib
| Fase | Event | Minggu |
| --- | --- | --- |
| **Onboarding** | Onboarding Session | Minggu 0 |
| **Learning Sprint** | Kickoff, Study Group x2, Industry Sharing | Minggu 1-2 |
| **Mini Project** | Kickoff, Workshop, Code Review, Sprint Review | Minggu 3-8 |
| **Collaborative Project** | Kickoff, Workshop, Portfolio Review | Minggu 9-18 |
| **Closing** | Demo Day, Batch Showcase | Minggu 17-18 |
