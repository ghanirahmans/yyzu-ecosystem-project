# Alur Kerja Proyek (Project Workflow)

Untuk memastikan proyek berjalan dengan tertib, terukur, dan menyerupai standar industri digital, YYZU menerapkan alur kerja yang terbagi menjadi sembilan tahapan utama:

```
Idea ──> Proposal ──> Planning ──> Team Setup ──> Development ──> Review ──> Demo ──> Documentation ──> Release/Archive
```

---

## Tahap 1: Idea (Ideasi)
Proses pencarian ide solusi, analisis masalah, serta riset awal kegunaan produk. Ide proyek dapat berasal dari kebutuhan internal komunitas, inisiatif member, maupun permintaan dari pihak luar/mitra industri.

## Tahap 2: Proposal (Penyusunan Proposal)
Ide yang telah disepakati dituangkan dalam proposal singkat yang menjelaskan:
- Masalah yang diselesaikan.
- Deskripsi fitur produk.
- Target pengguna.
- Teknologi yang direncanakan.
- Estimasi waktu pengerjaan kasar.

## Tahap 3: Planning (Perencanaan & Lingkup Kerja)
Penyusunan rencana pengerjaan proyek secara mendalam. Pada tahap ini, tugas-tugas dipecah (*task breakdown*), prioritas pengerjaan ditentukan, dan penentuan batas waktu (*milestones*) disepakati untuk menghindari pelebaran lingkup kerja (*scope creep*).

## Tahap 4: Team Setup (Pembentukan Tim)
Proses pembagian peran anggota tim yang terlibat. Tim ideal terdiri atas kolaborasi lintas peran (misal: 1 Product Manager, 1 UI/UX Designer, dan 2-3 Developers) dengan pembagian tugas yang jelas untuk masing-masing peran.

## Tahap 5: Development (Pengembangan Aktif)
Tahap pengerjaan teknis dimulai. Designer merancang wireframe dan visual interface di Figma, sementara Developer menulis kode pemrograman berdasarkan rancangan tersebut secara berkala menggunakan version control (seperti Git).

## Tahap 6: Review (Proses Peninjauan)
Karya yang sedang atau telah selesai dibangun wajib melewati proses peninjauan sebelum digabungkan. Developer melakukan *code review* melalui Pull Request, desainer antarmuka menguji kesesuaian visual, dan manajer produk memastikan fitur sesuai dengan kriteria yang ditentukan.

## Tahap 7: Demo (Demonstrasi Produk)
Setiap tim proyek mendemonstrasikan hasil aplikasi yang telah dibangun kepada sesama member dan mentor. Sesi ini ditujukan untuk memvalidasi fungsi aplikasi, melatih kemampuan presentasi teknis, serta mengumpulkan umpan balik dari audiens.

## Tahap 8: Documentation (Dokumentasi Akhir)
Setelah aplikasi berfungsi dengan baik, tim wajib menyusun dokumentasi akhir yang rapi, meliputi:
- Panduan pemasangan (*installation guide*).
- Dokumentasi API (jika ada).
- Struktur basis data atau arsitektur kode.
- Rangkuman tantangan dan pembelajaran tim selama proyek berlangsung.

## Tahap 9: Release / Archive / Continue (Peluncuran, Pengarsipan, atau Kelanjutan)
Keputusan akhir terhadap status proyek:
- **Release**: Aplikasi dideploy secara publik agar dapat digunakan oleh pengguna asli.
- **Archive**: Proyek diarsipkan dengan rapi jika cakupan kerja telah selesai seluruhnya dan tidak ada rencana pengembangan lanjutan.
- **Continue**: Proyek dilanjutkan ke fase/sprint berikutnya untuk pengembangan fitur tambahan.
